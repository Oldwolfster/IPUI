An essay on how we actually build

Progressive Anchoring
or, how I was wrong about software design for 45 years

Functional decomposition is a beautiful method. It's also a lie about how anyone actually writes code. Here's what really happens — and the method nobody named.

By Oldwolfster  ·  A field report from the other side of a paradigm shift

My favorite uncle moved to my home city when I was eight. He was ten years older than me, studying computer science, and the kind of nerd who actually wanted to babysit his weird little nephew so they could talk about computers. He handed me The C Programming Language — K&R, the white book — and I coded through it.

The thing he taught me, over and over, in every example and every problem we worked, was decomposition. Take the big thing. Break it into pieces. Each piece breaks into smaller pieces. Eventually you're looking at a few lines of code you can actually write. Build it up. Snap it together. Done.

I believed it. I built my career on it. I taught it to junior engineers for four and a half decades. If you can describe it, you can build it, I'd tell them, because you're just breaking off pieces of the description until it's all done!

Then, about a year ago, I sat down to automate my own coding process. The kind of project where you build a system that builds systems. To do that, I had to write down — precisely — how I actually decompose problems. I'd been preaching this method since 1981. Writing it down should have taken an afternoon.

It took a week. And at the end of the week I had to admit something so disorienting I was shell-shocked for six hours and stoned off my ass for the next twenty-four: I don't actually code that way. I never have. Nobody does.

This essay is what came out the other side. There's a method I've been using my whole career that I'd never named. It works with functional decomposition, not against it. And I think I finally understand why every project I've ever taught the "right way" still felt like I was sneaking some other process in through the back door.

The Method I Preached

Functional decomposition, the way I taught it, looks like this:

Start with one sentence that fully describes the system. Build a Pac-Man game. That's the root.

Break that sentence into 3–8 chunks that together cover 100% of the obligation. Nothing more. Nothing less. For Pac-Man:

  • Logic for credit granted / game start
  • Initialize
  • Play the game
  • End game when over
  • Check if high score and manage it

Each chunk must fully discharge its parent's responsibility. If "play the game" doesn't cover everything that happens during a Pac-Man round, your cut is wrong. Fix the cut.

Then you recurse. Play the game becomes check input → move Pac-Man → move ghosts → update score → check collisions. Each of those decomposes further. Eventually you bottom out at functions of a few lines, and you write them.

This method is gorgeous. It produces clean code. It produces clean documentation. It produces architecture diagrams that fit on one page and explain themselves. Niklaus Wirth wrote a paper called "Program Development by Stepwise Refinement" in 1971 that says essentially this, and the paper still holds up. I'd recommend it to anyone.

There's just one problem.

Functional decomposition only works when you already know where the seams are. And you almost never already know where the seams are.

Pac-Man is a fine example because Pac-Man has been built ten thousand times. The shape is known. We know what "play the game" means before we write a line of code. So the decomposition writes itself. It feels like the method is doing the work — but actually, history did the work. We're just re-implementing a shape humanity already discovered.

Try the same method on a system nobody has built before. Build a workbench for experimenting with neural networks. Now decompose. Four to eight chunks that fully cover the goal. Go.

You can't. There is no top sentence yet. You don't know what the system is until you build enough of it to find out. The act of decomposition presupposes an understanding that the act of building is supposed to produce.

This was the moment I'd been dodging for 45 years.

What I Actually Do

When I stopped trying to write down "the method I preach" and started watching what I actually do, here's what I found.

I want to build Pac-Man. I sit down at the machine. I don't decompose anything. I do this:

I get the screen into video mode. Two lines of inline assembly. The screen flips. Solid rock. I can stand on it.

But here's the part the textbooks skip. Those two lines are ugly. They look like this:

before
main():
    mov ax, 0013h    ; what a weird comment symbol... was it first? Maybe we are the weird ones.
    int 10h
    [50 more commands to draw Pac-Man]
    [20 more commands to poll the joystick]

Magic numbers. Cryptic primitives. Three things crammed into one place with no names declaring what any of them are. The code works, but it's ugly to read. Something in my gut says no, fix this first.

So before I drive the next anchor, I name the ugliness away:

after
main():
    set_video_mode()
    draw_pacman()
    check_input()

Same code. Same behavior. But now main() is a clean orchestration layer. The mess got pushed down one level into named children, and the names — set_video_mode, draw_pacman, check_input — convert mechanics into intent. The reader doesn't need to know what 0013h means anymore. The name tells them.

That's a decomposition. It happened on the very first anchor. It wasn't a "later phase." It happened immediately, the moment the code felt ugly to read.

And then I climb. Draw a Pac-Man sprite. Get him moving with the arrow keys. Add a board. Make him respect the walls. Each new anchor is working code, kept and built upon. Each new anchor, the moment it lands, gets the same treatment: does this feel ugly to read? If yes, pull out named helpers until it doesn't.

This is the method. I call it progressive anchoring, and the metaphor is climbing.

▲ Pac-Man (the target) ╱ ╱ ● collision rules ╱ ● board exists ╱ ● movement ╱ ● Pac-Man drawn ╱ ● video mode ─────────────────────────── you start here

You're climbing a mountain. The summit is the target — build Pac-Man. The target gives you direction, so you don't wander. But you can't see the route from the bottom. You don't know which holds are good and which ones crumble until you put weight on them.

So you drive an axe into the rock at the first place you can reach. Stand on it. Look at it — is it ugly? Clean it up. Then look around. Drive the next axe somewhere reachable from where you're standing. Stand on that. Clean it. Repeat.

You never have two unfinished anchors at once. The program is always running, just doing less than it eventually will. Every anchor is load-bearing. You don't throw any of them away.

Functional decomposition draws the map. Progressive anchoring picks where to walk first — and lets the working code reveal the seams. The thesis of this essay

The Burn Shows the Seams

Here is the part of the theory I want to dwell on, because it took me 45 years to see it and I think it's the load-bearing insight.

When you decompose code, you're cutting it along its seams. The question is: how do you know where the seams are?

Functional decomposition assumes you can see the seams from outside the code — that by thinking hard about the problem, you can draw the cuts before you write a line. This works for known shapes (Pac-Man, calculators, CRUD apps) because the seams are inherited from collective memory. It fails for novel shapes because you can't see what you don't yet understand.

Progressive anchoring works differently. You don't decide where the seams are. You write code, and the code burns.

The burn is a felt sense. You're coding along in a method, happy as a pig in shit, and at some point the method starts feeling wrong. Not broken — wrong. Maybe it's gotten long. Maybe it's doing more than its name promised. Maybe it's full of magic numbers. Maybe it's just ugly. Whatever flavor of wrong it is, your gut chirps: this needs to come apart.

That feeling is the seam revealing itself.

You couldn't have seen the seam from outside. The seam didn't exist until the code did. The act of writing the code generated the seam, and the burn is your perception detecting it. The decomposition isn't designed. It's discovered.

This is why progressive anchoring isn't just functional decomposition done bottom-up. A naive critic might say: "You're just decomposing the same way, you just write the code first." And the critic would be partly right — there is decomposition happening. But the trigger is fundamentally different. FD places seams from a model in the head. Progressive anchoring places seams where the working code says they belong. The seams come from evidence, not from prediction.

And — this is important — the same root mistake that breaks FD at the system level breaks pre-planned decomposition at any scale. If you tried to design the children of an anchor before writing the anchor, you'd be guessing about seams you can't see yet, just at smaller scale. The mistake isn't about size. It's about direction. Acting on a model in your head instead of evidence from the code is the same error whether the code is a whole system or twelve lines.

(There's a Knuth principle that has more teeth than ours, but the shape is the same. He named premature optimization as the root of all evil — making code fast before you know it needs to be fast. The companion principle is premature decomposition — splitting code before you know where the seams are. Both stem from the same root: acting on a model instead of evidence.)

Why Not Just Decompose First?

I want to give the critic their full hearing. The objection goes:

"Okay, but if you're going to end up decomposing anyway, why not just do it up front? Sketch out set_video_mode and draw_pacman and check_input before you write any code. Same destination, less rework."

This argument is seductive and wrong, and unpacking why is most of the point of this essay.

The destination looks the same — three named functions either way. But the path you took to get there determined whether those are the right three names. If I sketched them up front, before writing any code, those names came from my guess about how the program would naturally split. Sometimes I'd guess right. Often I'd guess wrong, and I'd carry that wrong guess forward for the rest of the project, fitting subsequent code into a structure that doesn't actually match the work.

You see this in real codebases all the time. A class named UserManager that's somehow grown to handle payment processing and email sending, because at some point someone needed those features and the existing structure didn't have anywhere obvious to put them, so they got jammed into UserManager. The class is doing three unrelated things because the name was wrong from day one — and the wrongness was invisible until enough code had been written to reveal it.

The wrong name codifies a misunderstanding of the shape. Every subsequent anchor has to fit a shape that isn't real. The compounding cost is enormous and almost entirely invisible at the point where it's incurred.

Now compare progressive anchoring. I write the raw asm. It works. It's ugly. I look at it and ask: what is this actually doing? Three things, it turns out — setting the video mode, drawing the character, polling input. I didn't have to predict those three things. The code told me there were three. I just named what was already there.

You don't design the seams. You detect them. The working code does the designing; your job is to listen.

The names that emerge from this process are honest, because they describe code that already exists. The names that emerge from top-down planning are guesses, because they describe code that doesn't exist yet. Honest names hold up under future change. Guessed names get warped to fit code they were never designed for.

This is also why progressive anchoring resists being automated. The burn — the felt sense of wrongness — lives in a human head, calibrated by experience, triggered by reading code against intent. No linter will ever detect it. No static analyzer. A method called draw_pacman() that polls the joystick is not syntactically broken. Only a reader who knows what draw_pacman is supposed to mean can notice the mismatch.

The Rhythm

So the actual practice isn't "anchor for a while, then decompose for a while." It's a continuous weave. The rhythm goes:

  1. Drive an anchor. Get working code.
  2. Look at it. Does it feel ugly, full, cryptic, biff-prone, or wrong in any way? If yes, name the ugliness away into helpers.
  3. Climb to the next anchor.
  4. Repeat.

The decomposition isn't a separate phase. It happens inside the rhythm, on every anchor, immediately when the burn fires. And the rhythm is fractal — the same motion runs at every scale.

Early on, main() is the container that fills, and the split happens there. Later, when you're inside draw_pacman() and it grows to 40 lines, that's the container that fills, and the split happens one level down. Later still, the helpers inside draw_pacman() grow and split themselves. Same rhythm. Same trigger. Different level.

Over many cycles, the system grows upward (new anchors at the top) and inward-downward (existing anchors decompose, their children eventually decompose too). The clean tree you see in finished documentation? It got assembled this way, layer by layer, burn by burn. Not designed top-down. Grown.

How They Differ

Functional DecompositionProgressive Anchoring
PurposeCreate structureCreate working structure
DirectionTop-downOutward from where you stand
Seams placed byDeveloper's predictionThe working code's burn
MetricCoverage of conceptConfirmation of truth
AssumesYou understand the shapeYou'll discover the shape
Best forRe-implementing known shapesBuilding shapes nobody knows yet

The "best for" row is the one that matters. Pac-Man is a known shape; functional decomposition handles it cleanly because the shape is in the collective memory of the field. A novel framework, a new neural network workbench, a piece of software that hasn't been built before — those are unknown shapes. Decomposition can't help with the discovery; anchoring can.

Almost every interesting project in our field is closer to the second category than the first. We just don't admit it.

How To Actually Do It

The hard part of progressive anchoring is choosing the next anchor and listening for the burn. Here are the heuristics I've developed over the years — most of them learned by failing to follow them and watching the consequences.

Aggressive when you're flowing, tiny when you're getting your ass beat Climbers do this instinctively. When the holds are solid and you can read the route, you make bigger reaches. When your forearms are pumped and the next hold is ambiguous, you shrink your moves. Hand to hand to hand, tiny, until you re-stabilize. Code is the same. If the last three things you tried worked first try, take bigger swings. If you've been debugging for an hour, the next anchor should be the smallest testable thing you can imagine.
Discomfort of any kind is the canary, and decomposition is the cure Ugly, complicated, easy to biff, doing several things, hard to follow — whatever makes the code feel wrong is the signal. Pull a named helper out. Even two lines deserve their own function if the name converts mechanics into intent. set_video_mode() containing mov ax, 0013h / int 10h is less ugly than the inline version, even though the LOC is identical. The decomposition doesn't reduce code — it converts mechanics into intent. Naming is the cure; the function is just where you put it.
Method length is a canary, not a rule Long methods chirp the canary. The chirp tells you to look, not to automatically split. Sometimes one objective is honestly thirty lines — TCP packet slicing, say — and that's fine. Leave it. But most long methods are doing more than their name promised, and the length is just how you noticed. The line count never decides. You decide, by looking. The line count just makes sure you bother to look.
Honest names enable the canary; vague names disable it The whole feedback loop depends on the function's name being an honest description of intent. draw_pacman() is honest — you can tell at a glance whether the body matches it. process_data() or handle_thing() is useless, because it's vague enough to "contain" anything. A vague name silences the canary. You literally lose the ability to detect when a function has overflowed its purpose, because there's no clear purpose to overflow.
Name the high-level slot early, even if it does almost nothing If you know you'll eventually have a parse_sdlc() function, call your read-the-file code parse_sdlc() from day one, even if all it currently does is read lines. The orchestration shape is what your future self needs. The implementation can grow into it.
When it gets ugly inside, rename and push down a level Your nice clean parse_sdlc() is now 40 lines and three nested loops. Don't try to clean it in place. Rename it to read_all_lines(), write a new clean parse_sdlc() that calls it, and let the mess live one level down where it's contained. This is the bricklayer's move — the rough work goes inside the wall, the smooth surface stays smooth.
If you can't reach the next anchor, decompose the gap When the climb stalls, the move is to step back and decompose the gap between where you are and where you're trying to be. This is functional decomposition arriving exactly when it's useful: as a tool for crossing a specific stuck place, not as a top-down master plan.
Abstractions are earned, not assumed No base class, no interface, no DRY pattern until the code proves it deserves the structure. The cost of a premature abstraction isn't writing it — it's that it codifies a misunderstanding of the shape, and now every subsequent anchor has to fit a shape that isn't real.
Keep everything that works Anchors are not prototypes. They are not exploratory spikes you throw away. The Pac-Man you see on screen at step 2 is the same Pac-Man in the finished game. The video-mode init from step 1 is still doing its job at the end. You climb through your anchors; you don't replace them.

The Top-Down Myth

The realization that sent me into 24 hours of stoned reorientation was this: nobody built software top-down. Ever. Not Knuth. Not Parnas. Especially not Carmack(props bro). Not me. The top-down story is what we tell after the fact, because the finished artifact happens to be hierarchical and clean.

Three things conspired to keep this hidden:

One. Documentation systematically erases the anchoring phase. David Parnas told us in 1986 to write up rational design processes as if they'd been planned that way, even when they weren't. He was right that the artifact should look clean. But the side effect is that every textbook, every architecture diagram, every "how we built this" talk shows the cleaned-up artifact and never the climbing process that produced it. The reader sees the map and assumes the cartographer drew it before walking the land. They didn't.

Two. Teaching examples are always known shapes. Every CS curriculum walks students through building Pac-Man, or a calculator, or a to-do list — things that have been built so many times the decomposition is collective memory. Students learn that "decomposition works" because they're only ever asked to decompose problems that have already been solved by humanity at large. They never get to feel the failure mode, because they never face an unknown shape.

Three. The anchoring phase feels like "just getting started" rather than "real engineering." So senior engineers do it for ten minutes, forget about it, and then teach the next four hours of decomposition as if that's where the work happens. The first ten minutes is where the understanding happens. The next four hours is just execution.

I taught it that way for 45 years.

What Changes Now

If you're a senior engineer reading this, the practical move is small but real: notice the phase you're in.

When you sit down to a problem and reach for the whiteboard and start drawing boxes, ask: do I actually know the shape of this system, or am I pretending I do? If you're re-implementing something well-understood — sure, decompose. If you're building something genuinely new, put the whiteboard away and drive your first axe into the rock. Get something running. Then look at it. Listen for the burn. Name what's there. Climb.

When you mentor junior engineers, teach them both methods. Tell them decomposition is real and necessary — but it's triggered by the burn, not by planning. Tell them which trigger to listen to based on whether they understand the problem yet. Don't let them suffer for ten years preaching one and using the other.

When you document a finished system, follow Parnas: write it top-down, present the clean artifact. But when you teach the process, be honest. Show the messy climb. Show the wrong turns. Show the moment a small ugly anchor became the foundation of something good.

I spent four and a half decades telling people that if they could describe a thing, they could build it. I still believe that. I was just wrong about which direction the describing flows. It doesn't flow top-down from a clean root sentence. It flows outward from working code — anchor by anchor, burn by burn, helper by helper — until enough of the system exists that the description finally becomes possible.

You don't describe the mountain and then climb it. You climb the mountain, and the description is what you can write when you reach the top.

A note on prior art. Almost nothing in this essay is original. David Parnas wrote about information hiding and design-for-change in 1972, and about faking rational design processes in 1986. Frederick Brooks wrote "plan to throw one away" in 1975. Alistair Cockburn described "walking skeletons." The Pragmatic Programmer talks about "tracer bullets." The XP community talks about spike solutions. Peter Naur wrote "Programming as Theory Building" in 1985 and may have come closest to the spirit of this idea.

What I'm hoping to contribute is a name for the specific move — anchors as load-bearing working code, kept and built upon, with decomposition triggered by the burn — and a reconciliation with the functional decomposition tradition that taught most of us to code. If you find yourself nodding while reading this, the chances are you already work this way and just hadn't named it either. That's the goal. Naming makes it teachable.
